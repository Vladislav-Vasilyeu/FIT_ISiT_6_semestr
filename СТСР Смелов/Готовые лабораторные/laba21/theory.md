# Задание 04 - HTTP-аутентификация

## 9. Три вида http-аутентификации, описанные в RFC

Три основных вида HTTP-аутентификации:

1. **BASIC Authentication** (Базовая аутентификация)
   - Описана в RFC 2617 и RFC 7617
   - Учетные данные кодируются в Base64

2. **DIGEST Authentication** (Дайджест-аутентификация)
   - Описана в RFC 2617 и RFC 7616
   - Использует хеширование вместо передачи пароля в открытом виде

3. **Bearer Token Authentication** (Аутентификация по токену)
   - Описана в RFC 6750 и RFC 7235
   - Использует токены (JWT, OAuth и т.д.)

---

## 10. Схема BASIC-аутентификации

**BASIC (Базовая) аутентификация** - самая простая схема HTTP-аутентификации.

**Принцип работы:**

1. Клиент отправляет запрос без учетных данных
2. Сервер ответствует с кодом 401 (Unauthorized) и заголовком:
   ```
   WWW-Authenticate: Basic realm="Protected Area"
   ```

3. Клиент кодирует учетные данные в формате `username:password` в Base64
4. Клиент повторно отправляет запрос с заголовком:
   ```
   Authorization: Basic base64(username:password)
   ```

5. Сервер декодирует Base64, проверяет учетные данные и предоставляет доступ

**Пример:**
- Учетные данные: `admin:password123`
- В Base64: `YWRtaW46cGFzc3dvcmQxMjM=`
- Заголовок: `Authorization: Basic YWRtaW46cGFzc3dvcmQxMjM=`

**Недостатки:**
- Пароль кодируется, но не шифруется (легко декодировать)
- Требует HTTPS для безопасности

---

## 11. Схема DIGEST-аутентификации

**DIGEST (Дайджест) аутентификация** - более безопасная схема, использующая хеширование.

**Принцип работы:**

1. Клиент отправляет запрос без учетных данных
2. Сервер ответствует с кодом 401 и заголовком:
   ```
   WWW-Authenticate: Digest realm="Protected", nonce="...", qop="auth"
   ```
   - `realm` - область защиты
   - `nonce` - уникальное значение сервера (число, используемое один раз)
   - `qop` - качество защиты (auth, auth-int)

3. Клиент вычисляет хеш (обычно MD5):
   ```
   response = hash(username:realm:password):nonce:hash(method:uri)
   ```

4. Клиент отправляет заголовок:
   ```
   Authorization: Digest username="admin", realm="Protected", 
                  nonce="...", uri="/resource", 
                  response="...", opaque="...", qop=auth, nc=00000001, cnonce="..."
   ```

5. Сервер проверяет хеш, используя известный пароль

**Преимущества:**
- Пароль не передается в открытом виде
- Защита от replay-атак (благодаря nonce и nc)
- Не требует HTTPS

**Недостатки:**
- Сложнее реализовать
- Требует сохранения паролей в открытом виде на сервере

## 12. Схема FORMS-аутентификации

**FORMS (Форменная) аутентификация** - не является стандартом HTTP, но широко используется в веб-приложениях.

**Принцип работы:**

1. Пользователь заполняет HTML-форму с username и password
2. Форма отправляется на сервер методом POST (обычно с HTTPS)
3. Сервер проверяет учетные данные:
   - Сравнивает с БД или системой аутентификации
   - Проверяет пароль (обычно хеш пароля)

4. При успешной аутентификации сервер создает сеанс:
   - Генерирует `Session ID` (уникальный идентификатор)
   - Сохраняет информацию о сеансе на сервере
   - Отправляет `Session ID` в cookie клиенту

5. Клиент автоматически отправляет cookie в каждом последующем запросе:
   ```
   Cookie: sessionid=abc123def456
   ```

6. Сервер проверяет Session ID и разрешает доступ

**Пример форма:**
```html
<form method="POST" action="/login">
  <input type="text" name="username" />
  <input type="password" name="password" />
  <button type="submit">Войти</button>
</form>
```

**Преимущества:**
- Гибкая и удобная для пользователя
- Можно применить CSRF-защиту
- Поддержка "Запомнить меня"
- Полный контроль над внешним видом и поведением

**Недостатки:**
- Не является стандартом HTTP
- Требует поддержки cookies
- CSRF-уязвимость при неправильной реализации

## 13. Почему HTTP-аутентификация в чистом виде не является надежной?

HTTP-аутентификация имеет следующие уязвимости:

### 1. **Передача по незащищенному каналу**
- Без HTTPS данные передаются в открытом виде
- Base64 легко декодируется (это не шифрование)
- Возможна перехват пароля

### 2. **Man-in-the-Middle (MITM) атаки**
- Злоумышленник может перехватить запрос и выдать себя за сервер
- Может подделать заголовок WWW-Authenticate

### 3. **Replay-атаки**
- В BASIC-аутентификации нет защиты от повторной отправки запроса
- Злоумышленник может использовать перехваченный заголовок Authorization несколько раз

### 4. **Невозможность выхода (logout)**
- HTTP-аутентификация stateless (без состояния)
- После входа браузер хранит учетные данные и отправляет их в каждом запросе
- Невозможно "выйти" без перезагрузки браузера или очистки cookies

### 5. **Отсутствие контроля над сеансом**
- Нет возможности управлять активными сеансами
- Нет возможности блокировать конкретный сеанс сервером

### 6. **Ограничения метода BASIC**
- Пароль хранится в памяти браузера
- Передается с каждым запросом

### 7. **Ограничения метода DIGEST**
- Требует хранения пароля в открытом виде на сервере
- Не защищает от других типов атак (XSS, SQL injection и т.д.)

### **Решение:**
Для надежной аутентификации необходимо использовать:
- **HTTPS** (всегда!)
- **Session-based аутентификация** (FORMS) с безопасным хранением сеансов
- **OAuth 2.0 / OpenID Connect** для третьих приложений
- **Хеширование паролей** (bcrypt, scrypt, PBKDF2)
- **CSRF-защита** при использовании форм
- **Двухфакторная аутентификация (2FA)**
