

## 9. Определение «Интернет»

**Интернет** — это глобальная децентрализованная сеть компьютерных сетей, объединяющая миллионы частных, государственных, академических и корпоративных устройств по всему миру. Построена на стеке протоколов TCP/IP, обеспечивает передачу данных между любыми подключёнными узлами независимо от их физического местоположения.

---

## 10. Определение «Служба Интернет»

**Служба Интернет** — это функциональная возможность сети Интернет, предоставляемая пользователям через определённый протокол и инфраструктуру. Примеры служб:
- **WWW** (World Wide Web) — доступ к гипертекстовым документам через HTTP
- **Электронная почта** — передача сообщений через SMTP/POP3/IMAP
- **FTP** — передача файлов
- **DNS** — разрешение доменных имён в IP-адреса
- **P2P-сети** — прямой обмен данными между пользователями

---

## 11. Определение «Узел сети Интернет»

**Узел сети Интернет** — это любое устройство (хост), подключённое к сети и имеющее уникальный IP-адрес, способное отправлять, принимать или ретранслировать данные. Узлы бывают:
- **Оконечные** (клиенты): компьютеры, смартфоны, IoT-устройства
- **Промежуточные**: маршрутизаторы, коммутаторы, серверы DNS
- **Серверные**: предоставляют ресурсы и службы другим узлам

---

## 12. Определение «Клиент-серверное приложение»

**Клиент-серверное приложение** — это архитектурная модель распределённого приложения, где:
- **Клиент** — компонент, инициирующий запросы к серверу (обычно интерфейс пользователя)
- **Сервер** — компонент, обрабатывающий запросы, выполняющий бизнес-логику и управляющий ресурсами (БД, файлы, вычисления)

Взаимодействие происходит по сети через чётко определённый протокол. Сервер может обслуживать множество клиентов одновременно.

---

## 13. Определение «Сетевой протокол»

**Сетевой протокол** — это формализованный набор правил, определяющих формат, порядок и семантику обмена данными между устройствами в сети. Включает:
- Синтаксис: структура сообщений, кодирование
- Семантику: значение каждого раздела и действия при получении
- Синхронизацию: порядок обмена и тайминги

Примеры: TCP, UDP, HTTP, FTP, SMTP.

---

## 14. Основные свойства протокола HTTP

| Свойство | Описание |
|----------|----------|
| **Текстовый** | Сообщения передаются в виде читаемого текста (ASCII/UTF-8) |
| **Без состояния (stateless)** | Сервер не сохраняет информацию о предыдущих запросах клиента |
| **Протокол запрос-ответ** | Клиент всегда инициирует соединение, сервер отвечает |
| **Расширяемый** | Поддерживает заголовки для передачи метаданных |
| **Независимый от транспорта** | Работает поверх TCP (порт 80/443), но может использовать другие |
| **Поддержка методов** | GET, POST, PUT, DELETE, PATCH и др. для разных операций |
| **Версионность** | HTTP/1.0, HTTP/1.1, HTTP/2, HTTP/3 с разными возможностями |

---

## 15. Состав HTTP-запроса

```
GET /index.html HTTP/1.1          ← Стартовая строка (метод, URI, версия)
Host: example.com                 ← Заголовки (key: value)
User-Agent: Mozilla/5.0
Accept: text/html
Content-Type: application/json    ← Тип тела (для POST/PUT)
Content-Length: 45
Authorization: Bearer token
                                  ← Пустая строка (разделитель)
{"login": "user", "pass": "123"}  ← Тело сообщения (опционально)
```

**Компоненты:**
1. **Стартовая строка** — метод, путь, версия протокола
2. **Заголовки** — метаданные о запросе и клиенте
3. **Пустая строка** — разделитель
4. **Тело** — данные (для POST, PUT, PATCH)

---

## 16. Состав HTTP-ответа

```
HTTP/1.1 200 OK                   ← Статусная строка (версия, код, текст)
Content-Type: text/html; charset=utf-8  ← Заголовки
Content-Length: 1234
Server: nginx/1.18.0
Set-Cookie: session=abc123
Date: Mon, 12 Feb 2024 10:00:00 GMT
                                  ← Пустая строка
<!DOCTYPE html>...                ← Тело ответа (HTML, JSON, файл и т.д.)
```

**Компоненты:**
1. **Статусная строка** — версия, код состояния (200, 404, 500), пояснение
2. **Заголовки** — метаданные о ответе и сервере
3. **Пустая строка**
4. **Тело** — запрошенный ресурс

## 17. Определение «Web-приложение»

**Web-приложение** — это клиент-серверное приложение, в котором:
- **Клиент** работает в браузере (HTML/CSS/JS)
- **Сервер** обрабатывает логику и данные
- **Взаимодействие** происходит через HTTP/HTTPS

Пользователю не требуется установка — достаточно URL. Примеры: Gmail, Google Docs, интернет-банкинг.

---

## 18. Определения «Frontend» и «Backend»

| Термин | Определение | Технологии |
|--------|-------------|------------|
| **Frontend** | Клиентская часть приложения, видимая пользователю. Отвечает за интерфейс, взаимодействие и отображение данных. | HTML, CSS, JavaScript, React, Vue, Angular |
| **Backend** | Серверная часть, выполняющая бизнес-логику, работу с БД, аутентификацию, вычисления. Невидима пользователю напрямую. | Node.js, Python, Java, PHP, C#, Go, базы данных |

**Связь:** Frontend делает HTTP-запросы к Backend, получает данные (обычно JSON) и отображает их.

---

## 19. Определение «Кроссплатформенное приложение»

**Кроссплатформенное приложение** — программное обеспечение, способное работать на различных операционных системах или платформах без изменения исходного кода или с минимальными адаптациями.

**Подходы:**
- **Веб-приложения** — работают в любом браузере
- **Node.js** — один код на Windows, Linux, macOS
- **Electron** — десктопные приложения из веб-технологий
- **React Native / Flutter** — мобильные приложения для iOS и Android

---

**Пояснение:**
1. Пользователь взаимодействует с интерфейсом в браузере

4. Серверное приложение выполняет бизнес-логику, обращается к БД
5. Результат возвращается клиенту в формате JSON/HTML
6. При необходимости сервер обращается к внешним API

## 22. Понятие «Асинхронная операция»

**Асинхронная операция** — это операция, которая выполняется независимо от основного потока выполнения программы, не блокируя его. При запуске такой операции программа не ждёт её завершения, а продолжает работу. По завершении операции вызывается callback или разрешается Promise.

**Преимущества:**
- Неблокирующий ввод-вывод
- Возможность выполнять другие задачи во время ожидания
- Эффективное использование ресурсов

**Пример:** Запрос к серверу занимает 500мс — вместо ожидания программа может обрабатывать UI или другие запросы.

## 23. Асинхронные запросы: XMLHttpRequest vs Fetch

### XMLHttpRequest (устаревший, но функциональный)

```javascript
const xhr = new XMLHttpRequest();
xhr.open('GET', 'https://api.example.com/data', true); // true = асинхронно

xhr.onreadystatechange = function() {
    if (xhr.readyState === 4 && xhr.status === 200) {
        const data = JSON.parse(xhr.responseText);
        console.log(data); // Обработка результата
    }
};

xhr.onerror = function() {
    console.error('Ошибка запроса');
};

xhr.send(); // Отправка, не блокирует код
console.log('Это выполнится ДО получения ответа');
```

**Принцип:** Создаётся объект → настраивается метод и URL → назначаются обработчики событий → отправка → callback при получении ответа.

### Fetch API (современный стандарт)

```javascript
// GET-запрос
fetch('https://api.example.com/data')
    .then(response => {
        if (!response.ok) throw new Error('HTTP error');
        return response.json(); // Парсинг JSON
    })
    .then(data => console.log(data)) // Обработка данных
    .catch(error => console.error(error)); // Обработка ошибок

// POST-запрос с async/await
async function createUser(userData) {
    try {
        const response = await fetch('https://api.example.com/users', {
            method: 'POST',
            headers: { 'Content-Type': 'application/json' },
            body: JSON.stringify(userData)
        });
        const result = await response.json();
        return result;
    } catch (error) {
        console.error('Ошибка:', error);
    }
}
```

**Принцип:** Возвращает Promise → цепочка .then() или await → не блокирует выполнение → более чистый синтаксис.

| Характеристика | XMLHttpRequest | Fetch API |
|----------------|----------------|-----------|
| API | Событийный (onload, onerror) | Promise-based |
| Код | Многословный | Лаконичный |
| Поддержка | Все браузеры | Современные (IE требует полифилл) |
| Прогресс | Есть встроенная | Требует ReadableStream |
| Отмена | xhr.abort() | AbortController |

## 24. Основное назначение сервера Node.js

**Node.js** — это среда выполнения JavaScript на стороне сервера, построенная на движке V8 (Chrome).

**Назначение:**
- Создание масштабируемых сетевых приложений (веб-серверы, API)
- Разработка real-time приложений (чаты, игры, стриминг)
- Микросервисная архитектура
- Инструменты сборки и автоматизации (CLI)
- Serverless-функции

**Ключевая идея:** Использование JavaScript единообразно на клиенте и сервере (full-stack JS).

---

## 25. Основные свойства сервера Node.js

| Свойство | Описание |
|----------|----------|
| **Event-driven (событийно-ориентированный)** | Всё построено на событиях и callbacks |
| **Неблокирующий I/O** | Операции ввода-вывода не блокируют поток выполнения |
| **Single-threaded с event loop** | Один главный поток обрабатывает все запросы через цикл событий |
| **Высокая производительность** | Эффективно обрабатывает тысячи одновременных соединений |
| **Модульная система (CommonJS/ESM)** | Код организуется в модули через `require` или `import` |
| **NPM-экосистема** | Более 2 миллионов пакетов для расширения функциональности |
| **Кроссплатформенность** | Работает на Windows, Linux, macOS, ARM |
| **V8-движок** | Компиляция в машинный код, высокая скорость выполнения JS |
| **Потоки (Streams)** | Эффективная обработка больших данных частями |
| **Cluster-модуль** | Возможность использования всех ядер CPU |

**Модель работы:**
```
┌─────────────────────────────────┐
│         Event Loop              │
│  ┌─────────────────────────┐    │
│  │   Очередь событий       │    │
│  │   (timers, I/O, etc.)   │    │
│  └─────────────────────────┘    │
│            │                    │
│            ▼                    │
│  ┌─────────────────────────┐    │
│  │  Thread Pool (libuv)    │    │
│  │  (файловая система,     │    │
│  │   DNS, тяжёлые задачи)  │    │
│  └─────────────────────────┘    │
└─────────────────────────────────┘
```
